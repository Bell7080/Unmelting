# 에나 동료 AI 설계 (Companion AI / RL)

Unmelting의 동료 캐릭터 **에나**에게 외부 API 없이 독자적 강화학습형 서포터 AI를 부여하기 위한 설계 문서.
구현 전 합의된 골격을 기록한다. **현재는 미구현**이며, 코드 진행 시 이 문서를 시작점으로 삼는다.

> 원형 기획은 `Unmelting_Game_Concept.md`, 구현 규칙은 `CLAUDE.md`가 우선한다.

---

## 1. 한 줄 정의

> **에나 = 플레이어를 관찰하는 일방향 서포터(AI 디렉터).**
> 보상은 *플레이어 본인이 자원관리·생존·고층을 **더** 잘하게 되는 향상*.
> 향상을 위해 *덜 돕는 절제*를 학습 → 밸런스 자동 보존.
> 의지 예산으로 가끔 개입하고 혼자 대사로 반응하며, 플레이어가 강해질수록 *수호자→응원단*으로 역할이 바뀐다.
> 귀여운 강아지의 탈을 쓴, 플레이어의 향상을 지켜보는 관리자.

---

## 2. 핵심 원칙 (절대 흔들지 않는 것)

1. **API 키 없음.** 모든 학습/추론은 로컬(빌드 시점 CI + 플레이어 브라우저)에서 돈다.
2. **일방향.** 에나는 플레이어의 *입력/말*을 이해하지 않는다. 오직 **게임 상태에 반응**만 한다. → 대화 트리·LLM 불필요.
3. **보상 = "생존"이 아니라 "향상".** "플레이어가 살았나?"가 아니라 "플레이어가 *스스로* **더** 잘하게 됐나?"를 보상으로 적는다. 이 한 글자("더")가 과보호 보모와 코치를 가른다.
4. **절제도 학습한다.** 과보호 = 노잼 = 낮은 보상. 학습이 알아서 도움을 아끼므로 밸런스가 보상 설계만으로 지켜진다.
5. **위력은 학습이 못 건드린다.** 학습은 *언제/어떤 종류로/어떤 말투로*만 정하고, 효과 수치(회복량·방패량·급소 보너스)는 상한 고정.

---

## 3. 3층 아키텍처

| 층 | 도는 곳 | 진짜/연출 | 역할 |
|---|---|---|---|
| **① 본능 바닥** | GitHub Actions (빌드 시점) | 진짜지만 *일반적* | 헤드리스 셀프플레이로 게임 자체를 못하진 않게. "타고난 재능". 너무 세지 않게 *유능한 정도*만. |
| **② 해금 연출** | 클라이언트 (스크립트) | **연출(가짜)** | 본능을 천천히 푸는 *체감 성장* + 강함을 늦게 여는 *밸런스 게이팅*. 미숙→능숙의 껍데기. |
| **③ 너에 대한 적응** | 플레이어 브라우저 (런타임) | **진짜 (영혼)** | 이 플레이어를 학습. "나를 안다"의 정체. 컨텍스추얼 밴딧/통계. |

- 본능(①)은 학습 시점에 *그 플레이어가 존재하지 않으므로* "너를 아는" 게 될 수 없다. 그건 ③의 몫.
- ②는 선택적 껍데기. ③의 진짜 적응만으로도 성장 체감은 나오며, ②는 그 위의 드라마/안전 바닥.

---

## 4. 학습이 도는 곳 (정적 사이트 / 추후 Steam)

- **GitHub Actions (빌드 시점)** = ① 오프라인 본능 학습 → 가중치 JSON을 정적 자산으로 번들. 카드게임은 상태가 작아 무료 CI 시간으로 충분. *Actions는 빌드 때 1회, 플레이어를 보지 않는다.*
- **브라우저 (런타임)** = ③ 플레이어별 온라인 적응 → `CompanionProfile`을 localStorage/IndexedDB에 저장. 서버·키 불필요.
- **저장 한계:** 현재 브라우저 저장은 기기/브라우저 로컬. 캐시 삭제·기기 변경 시 리셋. **추후 Steam 출시 시 Steam Cloud가 세이브 파일 동기화로 해결**(별도 백엔드 불필요).
- **전체 플레이어 통합 학습은 하지 않는다.** "개개인과 함께 성장"이 목표이므로 로컬 개인화가 정답.

---

## 5. 관찰(State) — 에나가 보는 것

런타임/시뮬 공통으로 추출하는 피처 벡터. (초기엔 핵심만, 점진 확장)

- **현재 위기:** 캐릭터 HP/방패, 불씨 티어(bright/dim/flickering/extinguished), 캔들 게이지.
- **예측되는 위협:** 미리보기 행(distance 1·2)의 강적/전방 3칸 그룹, 보스 공격까지 남은 턴(텔레그래프).
- **플레이어 행동 맥락:** 방금 밟은 카드 종류(거미줄 함정 등), 손패 사용/조합 빈도, 최근 선택의 모험성/수동성.
- **관계 기억(메타):** 이 플레이어가 자주 죽는 원인(불씨 고갈/함정/특정 층), 자주 안 쓰는 자원(방패/회복 꽃), 최근 런 대비 도달 층 추세.

## 6. 행동(Action) — 에나의 개입 "어휘"

의지 예산을 소모하는 개입 + 발화. 학습은 *어느 것을 언제* 고를지(+ 침묵)를 정한다.

- 클러치 방패(치사 피해 예측 시 `addShield`)
- 위기 회복(`heal`, 상한 고정)
- 0% 회피(적 반격 직전 1회 무효, `damageTaken=0`)
- 0% 급소(공격에 급소 보너스 — "아싸! 급소!")
- 성냥 보급(불씨 위기 시 `addHandCard`)
- 보물 포션(보물 개봉 시 회복 드롭 주입)
- 경고 발화 / 반응 바크 / 침묵

> 효과는 기존 시스템 훅 재사용: `Character.heal/addShield/addHandCard/gainCandle`, `ActionSystem`, `DropSystem`, `EmberSystem` 티어.

## 7. 보상(Reward) — 흐릿하므로 *간접 신호(proxy)* 로 학습

"몰입/향상"은 직접 못 잰다. 메타데이터 기반 proxy로 추론:

- 도달 층/턴이 *자기 최근 평균* 대비 상승 (= 더 깊은 모험 = 향상)
- 위기→회복의 굴곡(아슬아슬했지만 이김) = 최고의 몰입 순간 (+)
- 런 중도 이탈(분노 종료) ↓ (+)
- 손패·시스템 적극 활용 / 모험적 선택 (+)
- 다음 세션 복귀·세션 길이(로비 데이터) (+)
- **과보호로 난이도가 증발한 스톰프 (−)** ← 절제 학습의 핵심
- ⚠️ **time-on-task가 아니라 경험의 질(flow)을 목표로.** 조작적 스키너 상자 금지. "강아지" 프레임이 이 선을 지킨다.

---

## 8. 대사 시스템 — 작성(authored)이지 생성 아님

- **새 문장 생성 = LLM = API 필요 → 하지 않는다.**
- 대신 **깊은 풀을 손으로 작성**하고, 학습은 *언제/어느 것을/어떤 말투로* 고를지만 정한다.
- **말투 변주는 LLM 없이 가능:** 내부 *긴급도/기분*이 표면형(어미·문장부호·강조어·길이)을 변조.
  - 낮은 긴급도 → "불씨가… 약해지고 있어…"
  - 높은 긴급도 → "얼른 불을 켜야 해! 빨리!"
- "대사창을 자기 마음대로 띄운다" = **말하기 vs 침묵 자체가 학습된 행동.** 잔소리(과발화)엔 페널티.

### 대사 3티어
1. **이벤트 바크(가장 단순):** 특정 사건 직결. 예) 거미줄 함정 밟음 → "윽! 거미줄은 싫은데…", 처치 → "아싸!". 사건→풀 매핑 + *얼마나 자주 떠들지*만 학습.
2. **상황 코멘트:** 위기/예측 위협 반응. "큰 게 와… 조심해."
3. **클러치 대사:** 개입과 동반하는 결정적 한마디. "아직 끝낼 수 없어!"

- 로비↔인게임 메타 대사("이번엔 힘드네…", "다음엔 잘할 수 있을 거야")는 `CompanionProfile`을 읽어 라인 선택.

## 9. 학습 방식 — 컨텍스추얼 밴딧 권장

| 방식 | 학습 대상 | 평가 |
|---|---|---|
| **컨텍스추얼 밴딧 + 통계 (권장)** | "이 맥락에서 어떤 개입이 이 플레이어 향상에 기여했나" | 흐릿한 보상에 강함·안정·해석가능·작은 JSON |
| Tabular Q | 추상 위기상태의 개입 가치 | 가능, NN 없음 |
| 딥 RL 실시간 | 정책 전체 | 샘플 부족·불안정·연출 깨짐 → **비권장** |

- 변주("많이/적게/운 좋은 각성")는 **확률적 정책(stochastic policy)** 으로 자연 발생.
- "각성" = 드물게 터지는 내부 기분 급상승(일시 의지 환급/증폭) — 연출된 드라마, 예산 틀 안.

## 10. 밸런스 안전장치

- **의지/유대 예산:** 한 런의 총 도움량 상한. 단 *언제·얼마나 쓸지는 학습이 자유 배분*(방아쇠 아님, 예산).
- **효과 수치 상한 고정.** 학습은 위력 못 건드림.
- **절제 보상.** 과보호엔 (−).
- **역할 전환:** 초반=수호자(자주 구함), 후반(플레이어 능숙)=응원단(덜 구하고 플레이어 본인의 플레이를 칭찬·연출). 힘이 아니라 *역할*이 바뀐다.

## 11. 데이터/저장

- **`CompanionProfile`**(캐릭터 id별 분리): 관계 기억, 밴딧 가중치, 누적 통계, 대사 톤 상태.
- localStorage/IndexedDB → 추후 Steam Cloud.
- 캐릭터 추가 시 각자 개입 어휘 풀 + 대사 세트 + 보상 가중치만 갈아끼우면 성격 분리.

## 12. 구현 로드맵 (순서 주의)

> **GitHub Actions 학습은 *마지막* 단계다.** 학습할 환경·상태·보상·런타임 훅이 먼저 있어야 자동화할 게 생긴다.

1. **(본 문서) 설계 박제.** ✅
2. **런타임 스캐폴딩 먼저.** `CompanionSystem`(의지 예산 + 휴리스틱/규칙 정책 + 관찰 로깅 + 대사 훅)을 인게임에 통합. 학습 0이어도 *에나가 존재하고 반응*하게. → 캐릭터를 체감하고, state/action/reward를 코드로 확정.
3. **헤드리스 환경 정비.** 엔진 로직을 UI 없이 Node에서 돌려 셀프플레이. *이미 `EventSpawn.sim.test.ts`가 수천 회 헤드리스 시뮬을 돌리므로 절반은 깔려 있음.* 시스템(`GameState/ActionSystem/...`)은 DOM 비의존.
4. **로컬 오프라인 학습(①).** 셀프플레이로 본능 바닥 학습 → 가중치 JSON. *CI 아닌 로컬에서 먼저* (빠른 반복).
5. **가중치 적용.** 런타임 정책을 휴리스틱 → 학습 가중치로 스왑.
6. **온라인 개인화(③).** 브라우저 밴딧 + `CompanionProfile` 저장. 진짜 "너와 함께 성장".
7. **GitHub Actions 자동화 (마지막).** 4번 파이프라인이 검증되면 릴리스마다 재학습 → 본능 바닥이 버전업마다 진화.

> 참고: ③ 온라인 개인화는 Actions가 *전혀 필요 없다*(브라우저에서 돈다). Actions가 쓰이는 건 ① 본능 학습뿐. 상태가 작으면 *휴리스틱 + 온라인 밴딧만으로 충분할 수도* 있어, 무거운 오프라인 NN 학습은 필요할 때만.

## 13. 미정 / 결정 필요

- 본능 바닥(①)을 얼마나 강하게? (유능한 정도 권장 — 너무 세면 성장 여지·밸런스 손상)
- 첫 버티컬 슬라이스 개입 1~2종 선정 (불씨 경고 / 전방 강적 대비 후보)
- 의지 예산 회복·소모 곡선
- 변주/각성 강도 (얌전한 보조 ↔ 가끔 판을 뒤집는 드라마)
- 상태 피처 최소 집합 v1
